Use feature packages for capabilities and core only for narrow foundations. Interfaces hide meaningful behaviour; avoid pass-through abstractions. Two adapters make the present theme seam real.

TDD

One observable failing test → minimum implementation → refactor while green. Tests use caller interfaces, not private shape or internal mocks. Run the smallest task per cycle and all canonical gates before handoff.

Dependencies

Record non-trivial owned behaviour, why platform/stdlib is insufficient, KMP support and maintenance risk. Implement 1–2 trivial functions locally. Centralize every version in libs.versions.toml.

Documentation

Every docs/**/*.md has same-path HTML in the same commit. HTML preserves every MUST/DO-NOT, ID, command, path, status, risk, consequence and graph edge. Graphs need text/table equivalents. Run ./scripts/verify-doc-pairs.sh; humans review semantic parity.

Commits

Prefer build gates; tested design seam; CI; docs as separate explanatory commits. Keep intermediate states intentional, exclude build output, preserve unrelated work.

ADR-0001 guardrails

Never add account/sync ownership, nested tags, full-content extraction, a notification backend, dwell-time read heuristics, or Material-identical iOS UI. Reference accepted IDs instead of duplicating decisions.

ADR-0002 guardrails

  • Use Navigation 3 with concrete @Serializable, typed and unique keys under sealed AppNavKey : NavKey.
  • No string routes, Any stacks, mutable domain objects or display copy in keys.
  • Features emit callbacks/events; they never receive a navigator or mutate the stack.
  • Android owns NavDisplay; iOS renders the same runtime state through its adapter without a parallel graph.
  • Add visible and assistive copy to resources in the first slice: shared generated Compose resources, native resources only for genuine platform copy.
  • Run build-contract audits before release. Hard-coded production UI copy blocks release.

Handoff evidence

Report commit IDs, exact command results, real platform builds, toolchain limits, accessibility/device risks, and the next smallest test-first feature slice.
