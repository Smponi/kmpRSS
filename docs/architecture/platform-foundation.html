Platform experience foundation

Rule: share stable meaning and behaviour; let platforms own rendering and interaction. ReaderTheme is the small feature interface. Android and iOS are real adapters. Material remains Android-only; iOS never receives a Material look through shared code. Scope excludes onboarding feature UI.

Key identity: singleton destinations are data object; entity destinations are data classes containing stable IDs such as feedId/entryId. Mutable domain objects, copy/resources, callbacks and platform objects are forbidden. Add a stable instanceId only when simultaneous instances of the same entity need independent entry state; do not add random IDs otherwise. Equality means the same logical navigation entry, so released key types/fields evolve deliberately.

Navigation 3 Runtime supports Kotlin/Native, but the AndroidX Navigation 3 UI artifact is not assumed to be KMP UI. Non-Android restoration uses the explicit serialization configuration required by the selected stable version. Tests cover equality/uniqueness, serialization round-trips, restoration and exhaustive entry resolution. References: overview, state saving, KMP release notes.

Localizable presentation

Shared visible and assistive copy lives in commonMain/composeResources/values/strings.xml and is loaded with generated Res.string accessors plus stringResource/getString. Locale variants use qualified values-<locale> directories. Genuinely platform-only copy uses native resources without accidentally forking shared wording.

The rule includes semantics/accessibility, errors, placeholders, empty states, plurals/formats, notifications and platform chrome. Navigation keys contain identity only, never copy. New production UI uses resources in its first tracer slice; hard-code cleanup is not deferred to release. References: resources, localization.

Token policy

Share FoundationSpacing/FoundationSize because intent is stable. Adapters supply different ReaderColors/ReaderTypography. Shape, blur/material, chrome and navigation are not shared. Admit a token only when both adapters consume the same meaning. Android uses dynamic color plus static fallback; experimental Expressive APIs stay centralized. iOS uses scalable sp typography. Liquid Glass belongs in native SwiftUI/UIKit controls; Compose blur is not called Liquid Glass, and older iOS needs an opaque fallback.

Tomorrow’s first slice

Define onboarding state in a feature package, then one Android and one iOS screen. Do not start with a pixel-identical shared composable. Load all copy and semantics through resources, emit a typed Navigation 3 event using a unique AppNavKey, and prove the primary 48dp action plus Reduced Motion variant in the same tracer cycle.

Non-goals

No onboarding steps/copy, parsing, persistence, account/sync, or ADR-0001 change. Simulator compilation is not device accessibility proof.
