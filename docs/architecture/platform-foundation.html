Platform experience foundation

Rule: share stable meaning and behaviour; let platforms own rendering and interaction. ReaderTheme is the small feature interface. Android and iOS are real adapters. Material remains Android-only; iOS never receives a Material look through shared code. Scope excludes onboarding feature UI.

Token policy

Share FoundationSpacing/FoundationSize because intent is stable. Adapters supply different ReaderColors/ReaderTypography. Shape, blur/material, chrome and navigation are not shared. Admit a token only when both adapters consume the same meaning. Android uses dynamic color plus static fallback; experimental Expressive APIs stay centralized. iOS uses scalable sp typography. Liquid Glass belongs in native SwiftUI/UIKit controls; Compose blur is not called Liquid Glass, and older iOS needs an opaque fallback.

The pure policy is tested through its interface; adapters read system values at the theme seam. Feature semantics/previews and device audits remain mandatory.

Tomorrow’s first slice

Define onboarding state in a feature package, then one Android and one iOS screen. Do not start with a pixel-identical shared composable. Prove the primary 48dp labeled action and Reduced Motion variant in the same tracer-bullet cycle.

Non-goals

No onboarding steps/copy, parsing, persistence, account/sync, or ADR-0001 change. Simulator compilation is not device accessibility proof.
