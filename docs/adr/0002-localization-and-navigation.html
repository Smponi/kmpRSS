ADR-0002 · Accepted · 23. Juli 2026 · PRD-005 / PRD-011 / PRD-014

Copy is a resource. A route is a key.

Lokalisierung und Navigation werden als frühe Kompatibilitätsflächen behandelt: kein String bleibt in UI-Code gefangen und keine Plattform erfindet ein eigenes Routensystem.

Entscheidung 1 · Alle App-Texte sind lokalisierbar

Im Scope

Labels, Überschriften, Body, Buttons, Menüs, Placeholder, Accessibility-Namen/-Hints/-States/-Actions, Fehler, Loading/Empty States, Recovery, Formatwerte, Plurale, Notifications, Shortcuts, Widgets und Host-Chrome.

Ressourcen

Shared Compose: commonMain/composeResources/values/strings.xml, generierte Res.string-Accessor und stringResource/getString. Echte Plattform-Copy nutzt Android-/Apple-Ressourcen.

Nicht App-Copy

Remote Feed-Inhalt, URLs, interne IDs, diagnostische Logs und Test-Fixtures. Wie viele Sprachen ausgeliefert werden, ist separat; Externalisierung ist auch bei nur einem Locale Pflicht.

Entscheidung 2 · Navigation 3 ist autoritativ

Feature Event→Navigator→Navigation 3 Stack→Unique AppNavKey→Android NavDisplay / iOS Adapter
RegelVertrag
TypJeder Key ist ein konkretes @Serializable data object oder eine Data Class unter AppNavKey : NavKey.
IdentitätEquality bedeutet denselben logischen Entry. Singleton als Object; Entity-Ziel mit stabiler ID; instanceId nur für wirklich parallele Instanzen.
InhaltNur stabile, serialisierbare Identität. Keine Copy/Ressource, mutable Domain-Objekte, Callbacks, Services, Composables oder Plattformobjekte.
StackKeine String-Routen und kein Any-Stack. Features emittieren Events; ein testbarer Navigator mutiert den Stack.
ResolutionEin exhaustiver Entry Provider löst jeden Key genau einmal auf; unbekannte Keys scheitern in Development.
PlattformenAndroid nutzt NavDisplay. iOS rendert denselben Runtime-Stack Apple-gerecht und besitzt keinen parallelen Swift-/String-Graph.

Navigation 3 Runtime unterstützt KMP/Kotlin Native; das AndroidX UI-Artefakt wird nicht als KMP-UI vorausgesetzt. Non-Android-Restore nutzt die explizite Serialisierungskonfiguration der gewählten stabilen Version.

Verifikation vor Release

Localization

  • Keine sichtbaren/assistiven App-Literale in Production.
  • Base und deklarierte Locales besitzen kompatible Keys, Placeholder und Plurale.
  • Lang-/Pseudo-Text, maximale Schrift und relevantes RTL geprüft.
  • Audit-Evidence gespeichert; offener Literal blockiert Release.

Navigation

  • Serialisierungs-Roundtrip jedes Keys.
  • Dokumentierte Equality und eindeutige parallele Entries.
  • Backstack-Restore und exhaustive Entry-Auflösung.
  • Android und iOS rendern denselben autoritativen Zustand.

Konsequenzen

Gewinn: übersetzbare und accessibility-taugliche Copy, deterministische Navigation, kollisionsfreie Typen und testbare Feature-UI. Kosten: Ressourcen gehören in jeden ersten Feature-Slice; Resource-/Key-Namen werden Kompatibilitätsflächen; iOS braucht einen Display-Adapter; Release umfasst zusätzliche Audits.
